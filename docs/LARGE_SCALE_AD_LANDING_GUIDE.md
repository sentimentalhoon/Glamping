# 대규모 광고 집행용 랜딩 페이지 제작 가이드

## 목표
- 대규모 트래픽에서도 안정적으로 리드를 수집한다.
- 캠페인별로 메시지-오디언스 정합성을 높인다.
- 전환율(CVR)과 리드 품질(SQL)을 동시에 관리한다.

## 권장 구조 (광고 랜딩 전용)
1. 헤드라인 (5초 이해)
- 누구를 위한 어떤 가치인지 한 줄로 명확하게.
- 예: "프리미엄 글램핑 회원권, 연중 우선 예약과 자산형 혜택까지"

2. 신뢰 블록 (즉시 불신 해소)
- 실제 수치: 누적 고객, 재방문율, 응대 만족도
- 실사 이미지, 실제 후기, 언론/파트너 로고

3. 제안 블록 (상품/혜택)
- 개인/법인 멤버십을 카드로 분리
- 혜택 비교표(우선예약, 이용횟수, 양도 가능 여부)

4. 리스크 제거 블록 (의사결정 장애 제거)
- 환불/해지/상담 절차
- 방문투어 프로세스 및 소요시간

5. 단일 강한 CTA
- 페이지 내 CTA 문구/행동은 하나로 통일
- 예: "멤버십 상담 신청"

6. 폼 + 즉시 신뢰요소
- 필수 항목 최소화 (이름/연락처/관심유형/동의)
- 제출 직후 응답시간 명시 (예: 24시간 이내)

## 반드시 필요한 기술 구성

### 1) 추적 및 어트리뷰션
- UTM 유지: `utm_source`, `utm_medium`, `utm_campaign`, `utm_content`
- 저장 위치: hidden input + 서버 로그 + CRM 필드 동시 저장
- 이벤트 표준화
  - `page_view`
  - `cta_click`
  - `lead_submit`
  - `lead_qualified` (가능 시)

### 2) 리드 수집 파이프라인
- 브라우저 -> 서버 API(`/api/inquiries`) -> CRM/Webhook
- 실패 대비 재시도 또는 dead-letter 저장
- 운영 알림: 실패 즉시 Slack/문자

### 3) 성능/안정성
- 초기 렌더 2.5s 이내 목표 (LCP)
- 이미지/영상 경량화, 위 폴드 우선 최적화
- 광고 유입 시간대 피크 부하 대응 (서버 오토스케일/캐시)

### 4) 법적/정책
- 개인정보 동의 명시 + 처리방침 링크
- 허위/과장 표현 금지
- 캠페인 소재와 랜딩 문구 일치

## 캠페인 운영 방식 (권장)

### 1) 랜딩 분기
- 기본 랜딩: `/`
- 광고 전용 랜딩: `/lp/membership-a`, `/lp/membership-b`
- 각 랜딩은 오디언스별 메시지 차별화

### 2) A/B 테스트 우선순위
1. 헤드라인
2. 첫 CTA 문구
3. 폼 길이
4. 신뢰 블록 위치

### 3) 판단 지표
- 1차: 방문 대비 상담 제출율(CVR)
- 2차: 유효리드율(전화 연결/상담 진행)
- 3차: 계약 전환율(CPA/ROAS)

## 현재 프로젝트 기준 권장 추가 작업
1. UTM 파라미터를 폼 제출 payload에 포함
2. 개인정보처리방침/이용약관 페이지 생성 및 푸터 연결
3. 운영용 감사 로그(리드 ID, 타임스탬프, 캠페인 키) 추가
4. 광고 전용 경량 랜딩 1개를 별도 라우트로 신설

## 배포 전 기술 체크
- `npm run lint` 통과
- `npm run build` 통과 (현재는 `webpack` 기준)
- 실서버에서 폼 제출/웹훅 수신 end-to-end 테스트 통과
- 모바일(iOS/Android) 실기기 QA 통과
